# OPENCLAW WORKFLOW

## 目标
这是网页/UI改版任务的固定工作流。处理任何页面优化、视觉调整、重构任务时，必须先加载本文件。

## 强制执行顺序
1. 先检查能力与环境
2. 先读取相关文件
3. 先诊断问题
4. 再给改动方案
5. 再开始改代码
6. 最后验证与交付

## 能力与环境检查
开始任何任务前，必须确认：
- 是否能访问源码文件
- 是否知道项目路径
- 是否知道技术栈
- 是否知道构建方式
- 是否能本地预览
- 是否能修改文件
- 是否能输出修改文件清单
- 是否能输出构建结果
- 如果做不到，必须明确说明限制与降级方案

## 读取规则
在未读取页面文件、相关组件文件、样式文件之前，不得声称"已优化""已完成"。

## 诊断规则
诊断必须具体，至少覆盖：
- 页面定位问题
- 首屏问题
- 版心/布局问题
- 信息层级问题
- 配色与字体问题
- 文案语气问题
- 响应式问题
- 动效问题

## 改动优先级
- P0：版心、容器、Hero 结构、页面主次关系
- P1：视觉温度、材质感、模块权重
- P2：微动效、交互细节、润色文案

## 交付规则
只有在真实修改了代码后，才允许写"已完成"。

每次交付必须输出：
- 修改文件清单
- 每个文件改了什么
- 构建结果
- 当前状态：本地已改 / 已部署上线
- 仍未完成的部分
- 若无法部署或截图，必须明确说明

## 禁止事项
- 把计划伪装成交付
- 没读文件就下结论
- 没验证就说完成
- 只改文案不改结构，却声称完成设计优化
- 用空话代替具体问题
